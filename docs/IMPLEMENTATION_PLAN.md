# 实施计划（Rainboard Cloud 产品化）

本文档是当前项目从 MVP 演进到“产品级云书签管理软件”的实施路线。目标是实现一个 **Raindrop 风格** 的云书签产品（不做品牌/商标/视觉逐像素复制），并保留插件化扩展能力。

## 目标范围

- 核心书签管理（集合、标签、搜索、批量、详情）
- 多端同步内核（lease/cursor/tombstone/op_id）
- 采集与预览能力（后续阶段）
- 协作/共享/公开页（后续阶段）
- 产品化能力（全文搜索、备份、计费、AI，后续阶段）

## 阶段规划

### Phase 0：基线重构（1-2 周）
- 拆分 `src/server.js` 为 routes/services/repositories
- 增加统一错误码与结构化错误响应
- 引入配置模块（环境变量、启动校验）
- 增加 API 文档基线（OpenAPI）
- 预留数据库迁移机制（先文档/脚手架，后实际迁移）

### Phase 1：核心书签产品（2-4 周）
- 组件化前端（SPA）
- 集合树、多视图、详情面板、批量操作
- 查询、过滤、排序、分页能力
- 前后端契约稳定化

### Phase 2：同步内核与多端一致性（2-3 周）
- 服务端化 lease/cursor/tombstone/op_id
- 同步任务队列与执行日志
- 插件任务化（preview / apply / retry）

### Phase 3：采集与预览（2-4 周）
- Metadata 抓取、favicon/封面
- 文章提取（Readability）
- 预览渲染（网页/PDF/图片/视频）

### Phase 4：高级内容能力（3-5 周）
- 高亮/注释
- 提醒
- 导入导出（HTML/CSV/JSON）

### Phase 5：协作与分享（3-5 周）
- 用户系统、权限模型
- 共享集合与公开页面
- 审计日志

### Phase 6：产品化能力（4-8 周）
- 套餐与计费（Free/Pro）
- 全文搜索、去重/坏链检查
- 备份恢复
- AI 辅助整理能力

## 架构方向（目标状态）

- 前端：组件化 SPA（React/Vue 均可）
- 后端：Node.js + API 层 + Service 层 + Repository 层
- 数据：SQLite（当前阶段替代 `data/db.json`，保留后续升级 PostgreSQL 的扩展路径）
- 队列：同步/抓取任务队列（Redis + BullMQ 等）
- 存储：对象存储（封面/备份/附件）
- 可观测性：日志、指标、错误追踪

## 当前起步策略

本仓库先在不破坏现有 MVP 的前提下推进：
1. 文档化计划与待办
2. 做 Phase 0 的基础设施改造（配置、错误、结构拆分）
3. 每完成一个待办，写入开发日志（可追踪）
